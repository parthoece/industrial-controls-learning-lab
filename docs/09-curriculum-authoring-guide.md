# Curriculum authoring guide

Use this guide for new or revised lessons.

## Lesson contract

Every lesson must include:

- guiding question
- measurable objectives
- key terms
- diagram, equation, state model, or data model
- fundamentals-level explanation
- worked example
- common mistakes
- practical exercise
- knowledge checks with answers
- deep-study references
- exit criteria

## Writing style

- short paragraphs
- one idea per bullet
- tables for comparisons
- define symbols and units near equations
- prefer original diagrams
- avoid vendor marketing language
- explain abbreviations on first use
- separate fact, approximation, and design choice

## Technical quality

State:

- units
- coordinate frame
- sample time or timing assumption
- valid states and ranges
- failure behavior
- simulation or hardware status
- model limitations

## Reference quality

Prefer:

1. standards body or protocol organization
2. official vendor documentation
3. university or open textbook
4. primary technical paper

For each reference, include:

- why to read it
- relevant section
- general versus vendor-specific status
- access date when the source may change

Do not copy standards text, textbook prose, vendor diagrams, or licensed examples.

## Review checklist

- [ ] A new learner can answer the guiding question.
- [ ] The worked example is reproducible.
- [ ] The lesson links to code, a lab, or a design exercise.
- [ ] At least one failure is explained.
- [ ] Knowledge-check answers are correct.
- [ ] External links are current.
- [ ] Safety wording does not imply certification.
- [ ] Advanced topics are clearly deferred.
