# Job-Ready Portfolio Evidence

## Three-minute test

A hiring manager should quickly understand:

- which role the repository prepares you for
- what you implemented rather than merely read
- how software commands become physical motion
- how normal behavior is measured
- how failures are injected and diagnosed
- what is simulation-only and what, if anything, was hardware-tested
- where the future capstone continues the work

## Evidence checklist

| Evidence | Required in this learning repo |
| --- | :---: |
| Clear root README | Yes |
| Learning and role maps | Yes |
| Architecture and troubleshooting diagrams | Yes |
| Runnable mini-labs | Yes |
| Automated tests | Yes |
| Weekly experiment reports | Yes |
| Control response measurements | Yes |
| State-machine and interlock evidence | Yes |
| Motion/network troubleshooting checklist | Yes |
| Timing/jitter experiment | Yes |
| Manufacturing schema and OEE exercise | Yes |
| Capstone handoff specification | Yes |
| Demo video | Recommended after several labs are complete |

## Claims to avoid

- Do not claim real hardware commissioning when work was simulated.
- Do not call ordinary permission or interlock logic “functional safety.”
- Do not claim real-time guarantees from a general-purpose operating system timing experiment.
- Do not publish employer source code, machine drawings, customer data, credentials, or licensed vendor projects.
- Do not list tools or protocols without evidence showing what you did with them.

## Evidence folder convention

Learners may create a personal branch or fork and add:

```text
portfolio/
├── diagrams/
├── experiment-reports/
├── screenshots/
├── test-results/
└── demo-video.md
```

Generated databases and large binaries should not be committed. Prefer small CSV samples, images with captions, and links to externally hosted videos.
